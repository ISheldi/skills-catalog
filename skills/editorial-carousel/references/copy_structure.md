# Copy structure — как писать текст карусели

Это **копирайт-движок** скилла: прежде чем верстать слайды, напиши вирусный текст. Цель — донести мысль цепляюще, динамично, просто и применимо: так, чтобы поняли и подросток, и бабушка. Карусель — единственный формат, где читатель сам управляет темпом: листает, только если хочет. Значит **каждый слайд обязан давать причину листнуть дальше.**

Этот файл даёт структуру (что писать). Как это превращается в JSON и слайды — в `spec_schema.md`.

## Шаг 0 — откуда берём текст (спроси сразу)
Прежде чем писать, уточни **источник материала**:
- **С нуля по теме** — есть только тема, пишешь из своих знаний.
- **Из готового текста / статьи** — прислали текст или ссылку → вытаскиваешь суть, не сочиняешь.
- **Из транскрибации (видео / подкаст / эфир)** — прислали расшифровку → собираешь карусель из неё, сохраняя мысли и формулировки автора.
- **Смешанно** — материал + твои дополнения.

Если материал есть — **работай по нему**, не выдумывай факты сверх него. Если только тема — пишешь с нуля.

## Всегда давай 2 варианта
Ключевая привычка: из одного материала собери **2 разные карусели** — разный угол / наполнение / тип (например «инструкция» против «ошибки», или другой ракурс хука и обложки). Покажи **оба на выбор**; пользователь берёт один (или просит смешать). Только после выбора идём дальше (в конфигуратор).

## Почему карусели работают
- Алгоритм повторно показывает карусель тем, кто не долистал → бесплатные просмотры.
- «Сохранения» — самая ценная реакция, и каруселей сохраняют больше всего.
- Формат заставляет структурировать мысль → контент понятнее.

**Карусель проигрывает, когда:** обложка не остановила скролл; каждый слайд самодостаточен (нет причины листать); слишком много текста; нет нарратива — просто список фактов.

---

## Процесс (7 шагов)
1. **Выбери тип** из 6 ниже — какой подходит к материалу.
2. **Сформулируй обложку** — кристаллизация всей карусели в 1–2 строки. Не получается коротко → тема ещё не ясна.
3. **Набросай скелет** — название каждого слайда одним предложением. Проверь нарратив: каждый следующий логично вытекает из предыдущего?
4. **Пиши слайды по порядку** — заголовок + 2–4 предложения. Сначала быстро, потом сокращай.
5. **Финальный слайд** — конкретный вопрос или CTA. Это последнее, что запомнит читатель.
6. **Текст под постом** — хук + «листай →» (см. ниже).
7. **Проверь по чеклисту** перед тем, как отдавать в верстку.

## 6 типов каруселей

| тип | формула | почему работает |
|-----|---------|-----------------|
| 1. Инструкция | «Как [результат] за [N шагов / время]» | пришёл с задачей — ушёл с инструментом |
| 2. Ошибки | «N ошибок, которые [последствие]» | страх ошибки сильнее желания успеха |
| 3. Инсайт | «[Контринтуитивная мысль] — вот почему» | провоцирует несогласие, затем переубеждает |
| 4. До/После | «Было → Стало → Вот как» | трансформация — сильнейший нарратив |
| 5. VS | «[A] vs [B] — что выбрать?» | мозг любит сравнения, проще выбирать |
| 6. История | драматичный момент, разбитый на слайды | хочется узнать, чем кончилось |

### 1. Пошаговая инструкция
- Обложка: «Как [результат] за [N шагов / время]»
- Контекст (slide 2): кому нужно и почему стандартный подход не работает
- Шаги (по одному на слайд): название + 2–3 предложения + конкретный пример
- Финал: результат после всех шагов + CTA
- Оптимум: **5–8 шагов** (меньше 5 — поверхностно, больше 8 — теряется).

### 2. Список ошибок / «что не делать»
- Обложка: «[N] ошибок [в теме], которые [последствие]»
- Контекст: почему эти ошибки встречаются
- Слайды-ошибки: ошибка → **почему её делают** → как правильно (три элемента)
- Финал: «Какую из этих ошибок ты делал?» + CTA
- Важно: называй ПОЧЕМУ её делают — это снимает осуждение и создаёт близость.

### 3. Инсайт с доказательствами
- Обложка: спорное / неожиданное утверждение одной строкой
- Контекст: «большинство думает иначе, и вот почему»
- Аргументы (2–5): каждый с примером или данными
- Слайд-разворот: финальный аргумент, который всё переворачивает
- Финал: переформулированный тезис + «ты согласен?»
- Важно: провокация настоящая, не ложная — иначе раздражает.

### 4. До / После / Как
- Обложка: контраст «до» и «после» в одном заголовке
- «До»: конкретно, с болью (читатель должен узнать себя)
- Поворот: что изменилось
- «Как»: конкретные действия, приведшие к «после»
- «После»: результат — цифры, детали, ощущения
- Финал: «Хочешь так же?» / вопрос про их опыт

### 5. Сравнение / VS
- Обложка: «[A] vs [B] — что выбрать?»
- Когда A лучше — с примером
- Когда B лучше — с примером
- Нюансы / неожиданные сценарии
- Рекомендация: твой выбор и почему
- Финал: «А ты что используешь?»
- Важно: не давай «правильный» ответ с первого слайда — держи интригу.

### 6. История / кейс
- Обложка: самый драматичный момент («Я потеряла главного клиента. Вот что было дальше»)
- Экспозиция: кто, где, когда
- Разворачивание: конфликт, кризис
- Поворот: момент трансформации
- Результат: чем кончилось
- Финал: вывод + вопрос
- Важно: каждый слайд кончается «на полуслове» — это заставляет листать.

## Анатомия слайда

### Обложка — решает 80% успеха
Задача: остановить скролл. Потрать на неё ~30% времени.
- Заголовок 1–2 строки, крупно: провокация, вопрос или обещание конкретного результата.
- Подзаголовок (опц.): для кого / уточнение.
- **Без списков** — только заголовок. Визуально контрастна.

**Формулы заголовка:**
- «[Число] [чего] для [кого]» — «5 инструментов для тех, кто устал от совещаний»
- «Как [результат] без [типичного пути]» — «Как расти в LinkedIn без ежедневных постов»
- «[Провокация]» — «Нетворкинг не работает. Вот что работает вместо него»
- «[Вопрос]» — «Ты делаешь это в своих постах?»
- «[Обещание трансформации]» — «За 30 дней с 0 до 10K подписчиков. Разбираем как»

### Контентные слайды — правило одной мысли
Каждый слайд = **одна** идея. Не «несколько вещей», а одна чёткая мысль.
```
ЗАГОЛОВОК (1 строка — суть)
———
ТЕКСТ (2–4 предложения):
→ тезис
→ объяснение / пример
→ деталь / нюанс (опц.)
```
Не больше **4–5 строк** текста. Больше → раздели на два слайда. Белое пространство — это акцент, не пустота.

### Финальный слайд — конвертирует интерес в действие
Работает: конкретный вопрос в комментарии · «сохрани, чтобы вернуться» · короткий итог + CTA · ссылка (если уместно).
Не работает: «ставь лайк и подписывайся» · призыв на 5 строк · финал вообще без CTA.

## Текст под постом (caption)
Не дублирует слайды — это отдельный слой, трейлер.
```
ХУК (1–2 предложения) — та же мысль, что на обложке
→ контекст (2–3 предложения)
→ «Листай карусель →»
[читатель листает, возвращается]
→ подведение (1–2 предложения)
→ вопрос для комментариев
→ хэштеги (5–10 релевантных)
```
Длина: **100–400 знаков.**

## 6 принципов
1. **Обложка решает 80%.** Лучший контент не увидят, если обложка не остановила скролл.
2. **Каждый слайд продаёт следующий.** Держи нить нарратива.
3. **Меньше текста = больше внимания.** Перегруженный слайд пролистывают.
4. **Конкретность > общие слова.** «5 приёмов» > «несколько идей»; «за 30 дней» > «быстро».
5. **Финал — не конец, а начало.** Вопрос/призыв → комментарии и сохранения поднимают охват.
6. **Единый голос** от обложки до финала. Не прыгай с разговорного на официальный.

---

## Чеклист (перед версткой)
**Обложка:** останавливает скролл? · понятно за 1 сек, о чём карусель? · обещание результата или интрига?
**Слайды:** один слайд = одна мысль? · текст влезает без прокрутки? · после каждого есть причина листать? · есть конкретика/примеры/цифры? · структура слегка варьируется (нет монотонности)?
**Финал:** конкретный вопрос или CTA (не «ставь лайк»)? · написан текст под постом (хук + «листай →»)?
**Общее:** слайдов 5–10 (оптимум 7–8)? · голос единый? · карусель хочется сохранить «на потом»?

## Как нарратив ложится в типы слайдов спеки
Роли из методологии → `type` слайда в `spec.json` (см. `spec_schema.md`):

| роль в нарративе | тип слайда |
|------------------|------------|
| обложка | `cover` |
| контекст / экспозиция / «до» / «большинство думает» | `context` |
| шаг / ошибка / аргумент / «как» | `tip` (можно `badge` «01», `visual` или фон-фото) |
| цитата / провокация / резкий тезис | `quote` |
| финал — вопрос/CTA | `cta` |

Готовую текстовую структуру согласуй на шаге **Plan & approve**, затем перенеси в `spec.json`.
